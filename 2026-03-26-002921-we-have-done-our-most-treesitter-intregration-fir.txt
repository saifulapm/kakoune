⏺ Good. Next question:

  Where should the predicate evaluation logic live?

  Helix puts it in a wrapper around the query cursor (tree-house library).
  Neovim does it in C core. For our C++ integration, I see two options:

  A) In language_registry.cc — Parse predicates at query compile time (when
  grammar loads), store them per-pattern. Then evaluate in highlighters.cc /
  syntax_tree.cc when iterating matches. This matches the existing #set!
  pattern.

  B) In a new query_predicates.hh/cc — Dedicated module that wraps TSQueryCursor
   iteration, auto-filtering matches that fail predicates. All call sites
  (highlighters.cc, syntax_tree.cc, commands.cc) use the wrapper instead of raw
  ts_query_cursor_next_match/capture.

  I'd recommend A — it's simpler, keeps things additive, avoids a new
  abstraction layer, and follows the existing pattern where language_registry.cc
   already parses #set! predicates. We'd add a PredicateList per pattern to
  LanguageConfig and a small evaluate_predicates() free function.

  Which approach do you prefer?

❯ A                                                                             

⏺ Last question:

  Should #match? use C++ std::regex or a lighter approach?

  The queries use patterns like ^[A-Z_]+$, ^[a-z], ^__.*__$ — basic POSIX-style
  regex. Options:

  A) std::regex — Standard C++, no dependencies. Known to be slow for complex
  patterns but fine for these simple ones. We can cache compiled regexes
  per-pattern at load time.

  B) Kakoune's built-in Regex — Already in the codebase (regex.hh), uses a
  custom engine. Consistent with the rest of the editor.

  I'd recommend B — Kakoune already has a solid regex engine, avoids std::regex
  perf pitfalls, and keeps dependencies internal.

  Your preference?

❯ B                                                                             

⏺ Good choices. Here's the design:

  Design: Core Text Predicate Support

  Data Structures (in language_registry.hh)

  enum class PredicateType { Eq, NotEq, Match, NotMatch, AnyOf };

  struct QueryPredicate {
      PredicateType type;
      uint32_t capture_id;           // which capture to test
      String value;                  // string literal (for Eq/NotEq)
      Optional<uint32_t> capture_id2; // second capture (for capture-vs-capture
  Eq)
      Vector<String> values;         // for AnyOf
      Optional<Regex> regex;         // compiled regex (for Match/NotMatch)
  };

  // Per-pattern list stored in LanguageConfig
  using PatternPredicates = Vector<Vector<QueryPredicate>>;

  Parse Phase (language_registry.cc)

  At grammar load time, for each query (highlights, locals, injections,
  textobjects, indents):
  - Iterate ts_query_predicates_for_pattern() for every pattern
  - Parse predicate name → PredicateType
  - Extract capture ID and string/regex arguments
  - Compile regexes once via Kakoune's Regex
  - Store in PatternPredicates alongside existing InjectionPattern

  Evaluate Phase

  A free function:
  bool predicates_match(const PatternPredicates& preds,
                        const TSQueryMatch& match,
                        const Buffer& buffer);

  Called right after ts_query_cursor_next_match() / next_capture() in:
  - highlighters.cc — highlight loop and locals loop
  - syntax_tree.cc — injection detection
  - commands.cc — text objects, node selection

  If any predicate for the match's pattern fails → skip the match (continue the
  loop).

  Predicate Semantics

  ┌───────────────────────────┬───────────────────────────────┐
  │         Predicate         │             Logic             │
  ├───────────────────────────┼───────────────────────────────┤
  │ #eq? @cap "str"           │ node text == "str"            │
  ├───────────────────────────┼───────────────────────────────┤
  │ #eq? @cap @cap2           │ node text == node2 text       │
  ├───────────────────────────┼───────────────────────────────┤
  │ #not-eq? @cap "str"       │ node text != "str"            │
  ├───────────────────────────┼───────────────────────────────┤
  │ #match? @cap "regex"      │ Regex matches node text       │
  ├───────────────────────────┼───────────────────────────────┤
  │ #not-match? @cap "regex"  │ Regex doesn't match node text │
  ├───────────────────────────┼───────────────────────────────┤
  │ #any-of? @cap "a" "b" "c" │ node text in {"a","b","c"}    │
  └───────────────────────────┴───────────────────────────────┘

  Node text extracted via buffer coordinates from TSNode start/end.

  Design: Indent Algorithm Rewrite

  Core Function

  int compute_indent_for_line(const SyntaxTree& syntax_tree,
                              const Buffer& buffer,
                              LineCount line,
                              bool new_line);

  Returns the target indent level (in indent-width units) for the given line.
  Both tree_indent_cmd and tree_indent_newline_cmd call this.

  Algorithm (matching Helix's parent-walk)

  1. Find the innermost node at the line's byte position
  2. Walk up through parent nodes to the root
  3. At each node, collect indent captures that apply to this node
  4. Apply scope semantics (all vs tail) based on whether the node is first on
  its line:
    - Scope::All: if node is first on line → apply to current line, else → apply
   to line below
    - Scope::Tail (default for @indent): always apply to line below
  5. When crossing line boundaries (node and parent on different lines),
  finalize the current line's indentation and start accumulating for the next
  6. At root, return final net indent

  Indent Capture Types

  ┌─────────────────┬──────────────┬────────────────────────────────────────┐
  │     Capture     │   Default    │               Semantics                │
  │                 │    Scope     │                                        │
  ├─────────────────┼──────────────┼────────────────────────────────────────┤
  │ @indent         │ Tail         │ Sets indent=1 (unless indent_always >  │
  │                 │              │ 0)                                     │
  ├─────────────────┼──────────────┼────────────────────────────────────────┤
  │ @indent.always  │ Tail         │ Increments indent_always, clears       │
  │                 │              │ indent                                 │
  ├─────────────────┼──────────────┼────────────────────────────────────────┤
  │ @outdent        │ All          │ Sets outdent=1 (unless outdent_always  │
  │                 │              │ > 0)                                   │
  ├─────────────────┼──────────────┼────────────────────────────────────────┤
  │ @outdent.always │ All          │ Increments outdent_always, clears      │
  │                 │              │ outdent                                │
  └─────────────────┴──────────────┴────────────────────────────────────────┘

  The @indent vs @indent.always distinction: regular indent is capped at 1 per
  node (multiple @indent on same node don't stack). @indent.always stacks and
  overrides regular indent.

  injection handling. Stored as Vector<IndentScope> indexed by pattern, on
  LanguageConfig.

  - @align / @anchor — parameter alignment (orthogonal feature)

   predicates incorrectly.

   For large files, we risk exhausting the 256 match limit with distant
  captures.

  - @align / @anchor — affects parameter alignment in C, Julia
